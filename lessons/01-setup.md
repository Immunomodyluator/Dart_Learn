# Урок 1. Установка и инструменты

> Охватывает подтемы: 1.1 Установка SDK, 1.2 Dart CLI, 1.3 pub и pubspec.yaml, 1.4 Форматирование и линтеры

---

## 1. Формальное определение

**Dart SDK** — набор инструментов, предоставляемый Google, включающий:

- **Dart VM** — виртуальная машина с JIT-компилятором для разработки
- **dart2native / AOT-компилятор** — компиляция в нативный машинный код для продакшна
- **dart2js** — компиляция в JavaScript для браузера
- **dartfmt / dart format** — форматтер кода
- **dart analyze** — статический анализатор
- **pub** — пакетный менеджер

Уровень: **инфраструктура и tooling**. Знание инструментов определяет скорость разработки и качество кода ещё до написания первой строки логики.

---

## 2. Зачем это нужно

**Проблемы, которые решает SDK:**

- Единое окружение: нет разрыва между инструментом разработки и продакшн-рантаймом
- `dart format` устраняет споры о стиле: код форматируется автоматически по единому стандарту
- `dart analyze` находит ошибки до запуска (null safety нарушения, dead code, неверные типы)
- `pub` управляет зависимостями с воспроизводимым lockfile (`pubspec.lock`)

**Сценарии применения:**

| Сценарий | Инструмент |
|---|---|
| Flutter-разработка | `flutter` CLI (обёртка над Dart SDK) |
| Серверный Dart | `dart compile exe` → нативный бинарник |
| CLI-утилиты | `dart run`, `dart compile exe` |
| Pub-пакеты | `dart pub publish` |

## 3. Как это работает

### JIT vs AOT

Dart поддерживает два режима компиляции:

- **JIT (Just-In-Time)** — используется при `dart run`. Код компилируется в байткод, VM оптимизирует горячие пути во время выполнения. Быстрый цикл: изменил → запустил. Поддерживает hot reload во Flutter.
- **AOT (Ahead-Of-Time)** — `dart compile exe`. Код компилируется заранее в машинный код для целевой платформы. Нет VM overhead, холодный старт быстрее, размер бинарника ~5-10 МБ.

### pub и pubspec.yaml

`pubspec.yaml` — манифест проекта (аналог `package.json` в Node.js или `pom.xml` в Java):

```yaml
name: my_app
description: Example Dart application
version: 1.0.0
environment:
  sdk: '>=3.0.0 <4.0.0'       # Ограничение версии SDK

dependencies:
  http: ^1.2.0                 # Продакшн-зависимость

dev_dependencies:
  lints: ^3.0.0                # Только для разработки
  test: ^1.24.0
```

После `dart pub get` создаётся `pubspec.lock` — фиксирует точные версии всех транзитивных зависимостей. Этот файл **нужно коммитить** в приложениях, но **не в пакетах**.

### Линтеры и анализатор

Файл `analysis_options.yaml` в корне проекта управляет правилами:

```yaml
include: package:lints/recommended.yaml

linter:
  rules:
    prefer_const_constructors: true
    avoid_print: true

analyzer:
  errors:
    missing_required_param: error   # Превращает предупреждение в ошибку
```

## 4. Минимальный пример

```bash
# Установка SDK (Windows — через winget или chocolatey)
winget install Dart.Dart

# Создание нового консольного приложения
dart create my_app
cd my_app

# Запуск через JIT
dart run

# Компиляция в нативный бинарник
dart compile exe bin/my_app.dart -o build/my_app.exe

# Форматирование всего проекта
dart format .

# Статический анализ
dart analyze
```

Структура нового проекта, созданного `dart create`:

```
my_app/
├── bin/
│   └── my_app.dart       # Точка входа
├── lib/
│   └── my_app.dart       # Основная логика (библиотека)
├── test/
│   └── my_app_test.dart  # Тесты
├── pubspec.yaml
├── pubspec.lock
└── analysis_options.yaml
```

## 5. Практический пример

Сценарий: CLI-утилита для обработки JSON-файлов. Настраиваем окружение под реальный проект.

```yaml
# pubspec.yaml
name: json_processor
version: 1.0.0
environment:
  sdk: '>=3.0.0 <4.0.0'

dependencies:
  args: ^2.4.0         # Парсинг аргументов командной строки

dev_dependencies:
  lints: ^3.0.0
  test: ^1.24.0
```

```yaml
# analysis_options.yaml
include: package:lints/recommended.yaml

analyzer:
  exclude:
    - build/**
  errors:
    invalid_annotation_target: ignore   # Для кодогенерации

linter:
  rules:
    avoid_dynamic_calls: true    # Запрет вызовов на dynamic
    prefer_final_locals: true    # Предпочитать final для локальных переменных
```

```bash
# Рабочий цикл
dart pub get          # Установить зависимости
dart analyze          # Проверить ошибки
dart format .         # Отформатировать
dart test             # Запустить тесты
dart compile exe bin/json_processor.dart -o dist/json_processor
```

**Архитектурная корректность:** анализ и форматирование запускаются до тестов — это стандартная последовательность в CI. `dart analyze` в режиме `--fatal-infos` превращает все info-уровни в ошибки, что полезно для строгих проектов.

## 6. Что происходит под капотом

### JIT-запуск (`dart run`)

1. Парсер читает `.dart` файлы → строит AST
2. Кernal compiler компилирует AST → **Kernel IR** (`.dill` — бинарный формат промежуточного представления)
3. Dart VM загружает `.dill` → компилирует в нативный код через **Crankshaft/Turbofan**-подобный JIT
4. VM профилирует горячие методы → перекомпилирует их с агрессивными оптимизациями

### AOT-компиляция (`dart compile exe`)

1. Те же шаги 1-2 (AST → Kernel IR)
2. **AOT-компилятор** precompiles всё в нативный машинный код
3. Линкуется Dart runtime (GC, isolate scheduler) → единый бинарник
4. Нет VM overhead при запуске, нет JIT-профилирования в runtime

### pub resolve

`dart pub get` запускает **version solver** — SAT/backtracking алгоритм, который находит совместимый набор версий всех зависимостей, удовлетворяющий всем `pubspec.yaml` ограничениям в дереве зависимостей.

## 7. Производительность и ресурсы

**Где Dart эффективен:**
- AOT-бинарники стартуют за ~10-50ms (против JVM с ~200-500ms для простых app)
- `dart format` — инкрементальный, работает только с изменёнными файлами

**Узкие места:**
- JIT-запуск первый раз медленнее AOT (нет прекомпилированного кода)
- `dart pub get` с большим деревом зависимостей может занять несколько секунд при версионных конфликтах
- `dart compile js` (для web) генерирует большой выхлоп — нужен treeshaking через `dart compile js -O4`

## 8. Частые ошибки и антипаттерны

**1. Не коммитить `pubspec.lock` в приложениях:**
```bash
# НЕВЕРНО для приложений — другой разработчик получит другие версии
echo "pubspec.lock" >> .gitignore
```

**2. Использовать `any` версию зависимостей:**
```yaml
# НЕВЕРНО — непредсказуемые обновления
dependencies:
  http: any

# ВЕРНО — каретка означает "совместимое обновление"
dependencies:
  http: ^1.2.0
```

**3. Запускать `dart analyze` с ошибками:**
```bash
# Если analyze выдаёт ошибки — нельзя считать код рабочим.
# В CI: dart analyze --fatal-warnings
```

**4. Игнорировать `analysis_options.yaml`:**
Без настроенного анализатора теряется половина ценности статической типизации Dart.

## 9. Сравнение с альтернативами

| Аспект | Dart | Node.js/npm | Java/Maven | Python/pip |
|---|---|---|---|---|
| Форматтер | `dart format` (встроен) | Prettier (отдельно) | google-java-format | Black (отдельно) |
| Анализатор | `dart analyze` (встроен) | ESLint (отдельно) | Checkstyle | mypy/pylint |
| Холодный старт (AOT) | ~10-50ms | ~100-300ms | ~200-500ms | ~50-200ms |
| Бинарник | Один файл | Нет (node_modules) | JAR + JVM | Нет |
| Версионный конфликт | Один lockfile | package-lock.json | Maven dep tree | pip-compile |

**Преимущество Dart:** всё встроено в один SDK — нет зоопарка инструментов.

**Недостаток:** меньший выбор пакетов по сравнению с npm/Maven.

---

## 10. Когда НЕ стоит использовать Dart

- **Скриптование** (bash/Python удобнее для одноразовых скриптов)
- **Data science** — отсутствие экосистемы (NumPy, pandas, TensorFlow)
- **Командная разработка с уже установленным Java-стеком** — переход не оправдан только ради инструментов
- **WebAssembly** — компиляция Dart → WASM пока менее зрелая, чем Rust или C++

---

## 11. Краткое резюме

1. **SDK = всё в одном:** компилятор (JIT/AOT), форматтер, анализатор, пакетный менеджер — ничего не нужно устанавливать отдельно.
2. **JIT для разработки, AOT для продакшна** — два режима компиляции с принципиально разными характеристиками.
3. **`pubspec.lock` коммитится в приложениях** — воспроизводимые сборки; в пакетах — нет (чтобы не ограничивать пользователей).
4. **`dart analyze` — обязательная часть рабочего цикла**, а не опциональная проверка.
5. **`analysis_options.yaml` настраивается под проект** — стандарт `package:lints/recommended.yaml` покрывает большинство потребностей.
6. **AOT-бинарник — самодостаточный**: нет зависимости от установленного Dart на целевой машине.
7. **`dart create` — стартовая точка**: стандартная структура проекта экономит время и устанавливает правильные конвенции.
