# 16.2 Dependency Injection и управление зависимостями

## 1. Формальное определение

**Dependency Injection (DI)** — паттерн, при котором объект получает свои зависимости извне, а не создаёт их самостоятельно. Это инверсия контроля (IoC): вместо «я создаю то, что мне нужно» — «мне дают то, что мне нужно».

## 2. Зачем это нужно

- **Тестируемость** — легко подменить зависимость на мок.
- **Гибкость** — смена реализации без изменения потребителя.
- **Разделение ответственности** — класс не знает, как создаются его зависимости.
- **Управление жизненным циклом** — singleton, factory, lazy.

## 3. Виды DI

### Конструкторный DI (рекомендуемый)

```dart
// Зависимость передаётся через конструктор
class UserRepository {
  final ApiClient _client;
  final CacheService _cache;

  UserRepository(this._client, this._cache);

  Future<User> getUser(String id) async {
    final cached = _cache.get<User>('user_$id');
    if (cached != null) return cached;

    final user = await _client.get<User>('/users/$id');
    _cache.set('user_$id', user);
    return user;
  }
}

// Использование
final repo = UserRepository(ApiClient(), CacheService());

// Тест
final repo = UserRepository(MockApiClient(), MockCacheService());
```

### Инъекция через параметр метода

```dart
// Для единоразовых зависимостей
class ReportGenerator {
  String generate(List<Order> orders, {Formatter? formatter}) {
    final fmt = formatter ?? DefaultFormatter();
    return fmt.format(orders);
  }
}
```

### Service Locator (get_it)

```dart
import 'package:get_it/get_it.dart';

final getIt = GetIt.instance;

void setupDependencies() {
  // Singleton — один экземпляр на всё приложение
  getIt.registerSingleton<ApiClient>(ApiClient(baseUrl: 'https://api.example.com'));

  // Lazy singleton — создаётся при первом доступе
  getIt.registerLazySingleton<Database>(() => Database());

  // Factory — новый экземпляр каждый раз
  getIt.registerFactory<UserRepository>(
    () => UserRepository(getIt<ApiClient>(), getIt<Database>()),
  );
}

// Использование
void main() {
  setupDependencies();
  final repo = getIt<UserRepository>();
}
```

### Кодогенерация (injectable)

```dart
import 'package:injectable/injectable.dart';

@module
abstract class AppModule {
  @singleton
  ApiClient get apiClient => ApiClient(baseUrl: 'https://api.example.com');
}

@injectable
class UserRepository {
  final ApiClient _client;
  UserRepository(this._client);  // Автоматически инжектируется
}

@injectable
class AuthService {
  final UserRepository _repo;
  AuthService(this._repo);
}

// build_runner генерирует код регистрации
// dart run build_runner build
```

## 4. Паттерны жизненного цикла

```
┌──────────── Жизненный цикл ──────────────────────┐
│                                                    │
│  Singleton                                         │
│  ├── Один экземпляр на всё приложение             │
│  ├── Для: ApiClient, Database, Logger              │
│  └── getIt.registerSingleton<T>(instance)          │
│                                                    │
│  Lazy Singleton                                    │
│  ├── Создаётся при первом запросе                  │
│  ├── Для: тяжёлые объекты, не всегда нужные       │
│  └── getIt.registerLazySingleton<T>(() => T())     │
│                                                    │
│  Factory                                           │
│  ├── Новый экземпляр при каждом запросе            │
│  ├── Для: ViewModel, Controller, UseCase           │
│  └── getIt.registerFactory<T>(() => T())           │
│                                                    │
│  Scoped (продвинутый)                              │
│  ├── Экземпляр на «область» (запрос, экран)        │
│  ├── getIt.pushNewScope() / popScope()             │
│  └── Для: per-request state на сервере             │
│                                                    │
└────────────────────────────────────────────────────┘
```

## 5. Абстракции и интерфейсы для DI

```dart
// Абстракция (контракт)
abstract class UserRepository {
  Future<User> getById(String id);
  Future<void> save(User user);
}

// Реальная реализация
class ApiUserRepository implements UserRepository {
  final ApiClient _client;
  ApiUserRepository(this._client);

  @override
  Future<User> getById(String id) async =>
      await _client.get<User>('/users/$id');

  @override
  Future<void> save(User user) async =>
      await _client.post('/users', body: user.toJson());
}

// Мок для тестов
class MockUserRepository implements UserRepository {
  @override
  Future<User> getById(String id) async =>
      User(id: id, name: 'Test User');

  @override
  Future<void> save(User user) async {}
}

// Регистрация
getIt.registerFactory<UserRepository>(
  () => ApiUserRepository(getIt()),
);
```

## 6. Environments (dev/prod/test)

```dart
// С get_it
void setupDependencies({required String environment}) {
  if (environment == 'prod') {
    getIt.registerSingleton<ApiClient>(
      ApiClient(baseUrl: 'https://api.production.com'),
    );
  } else {
    getIt.registerSingleton<ApiClient>(
      ApiClient(baseUrl: 'https://api.staging.com'),
    );
  }
}

// С injectable (кодогенерация)
@Environment('prod')
@Injectable(as: ApiClient)
class ProdApiClient implements ApiClient { ... }

@Environment('dev')
@Injectable(as: ApiClient)
class DevApiClient implements ApiClient { ... }
```

## 7. Распространённые ошибки

### ❌ Service Locator как глобальная переменная

```dart
// Плохо — зависимость скрыта внутри метода
class UserService {
  Future<User> getUser(String id) {
    final repo = getIt<UserRepository>();  // Скрытая зависимость!
    return repo.getById(id);
  }
}

// Хорошо — явная зависимость через конструктор
class UserService {
  final UserRepository _repo;
  UserService(this._repo);

  Future<User> getUser(String id) => _repo.getById(id);
}
```

### ❌ Circular dependencies

```dart
// A зависит от B, B зависит от A → StackOverflow!
// Решение: выделить общую логику в третий класс
// или использовать ленивую инициализацию
```
