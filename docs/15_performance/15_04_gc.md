# 15.4 Память и сборщик мусора

## 1. Формальное определение

**Сборщик мусора (GC)** в Dart VM использует **поколенческий** алгоритм. Куча разделена на два поколения:

- **Young Space (New Generation)** — для короткоживущих объектов. Частые, быстрые сборки (scavenge).
- **Old Space (Old Generation)** — для долгоживущих объектов. Редкие, полные сборки (mark-sweep / mark-compact).

## 2. Зачем это нужно

- **Утечки памяти** — объекты, удерживаемые ссылками дольше необходимого.
- **GC-паузы** — Full GC может вызвать заметную паузу в приложении.
- **OOM** — исчерпание памяти приводит к крашу.
- **Производительность** — понимание GC помогает писать GC-friendly код.

## 3. Как работает GC в Dart

```
┌──────────── Куча Dart VM ─────────────────────────┐
│                                                    │
│  Young Space (New Generation)                      │
│  ├── Semi-space A (active)                        │
│  ├── Semi-space B (inactive)                      │
│  └── Scavenge: копирование живых из A в B         │
│      ~1 мс, частые сборки                         │
│                                                    │
│  Old Space (Old Generation)                        │
│  ├── Объекты, пережившие N scavenge'ей            │
│  ├── Mark-Sweep: помечаем живые, удаляем мёртвые  │
│  └── Mark-Compact: + дефрагментация               │
│      ~10-50 мс, редкие сборки                     │
│                                                    │
│  Продвижение (Promotion):                          │
│  Young → Old после нескольких выживаний            │
│                                                    │
└────────────────────────────────────────────────────┘
```

### Жизненный цикл объекта

```
Создание → Young Space → [Scavenge: жив?]
                          ├── Нет → освобождён
                          └── Да → копируется в другой semi-space
                                    → [N scavenge'ей пережито?]
                                       ├── Нет → остаётся в Young
                                       └── Да → продвигается в Old
                                                 → [Full GC] → освобождён
```

## 4. Утечки памяти

### Типичные причины

```dart
// ❌ 1. Забытые подписки на Stream
class MyWidget {
  late StreamSubscription _sub;

  void init() {
    _sub = eventStream.listen((e) => handle(e));
  }

  // Забыли отменить подписку!
  // void dispose() => _sub.cancel();
}

// ✅ Исправление
class MyWidget {
  late StreamSubscription _sub;

  void init() {
    _sub = eventStream.listen((e) => handle(e));
  }

  void dispose() {
    _sub.cancel();  // Обязательно!
  }
}
```

```dart
// ❌ 2. Замыкания, удерживающие контекст
class DataProcessor {
  final List<int> _hugeData = List.generate(1000000, (i) => i);

  Function getCallback() {
    // Замыкание удерживает ссылку на this (и _hugeData)
    return () => print(_hugeData.length);
  }
}
// Даже если DataProcessor больше не нужен,
// пока жив callback — _hugeData остаётся в памяти
```

```dart
// ❌ 3. Статические коллекции, растущие без ограничений
class Cache {
  static final Map<String, Object> _cache = {};

  static void put(String key, Object value) {
    _cache[key] = value;  // Никогда не очищается!
  }
}

// ✅ Используем LRU-кэш с ограничением
class LruCache<K, V> {
  final int maxSize;
  final _map = LinkedHashMap<K, V>();

  LruCache({this.maxSize = 100});

  V? get(K key) {
    final value = _map.remove(key);
    if (value != null) _map[key] = value; // Перемещаем в конец
    return value;
  }

  void put(K key, V value) {
    _map.remove(key);
    _map[key] = value;
    while (_map.length > maxSize) {
      _map.remove(_map.keys.first); // Удаляем самый старый
    }
  }
}
```

```dart
// ❌ 4. Timer, который не отменяется
Timer.periodic(Duration(seconds: 1), (timer) {
  updateUI();  // Удерживает ссылки
});
// timer.cancel() нигде не вызывается!
```

## 5. WeakReference и Finalizer

```dart
// WeakReference — не препятствует сборке GC
import 'dart:core';

class ImageCache {
  final _cache = <String, WeakReference<ImageData>>{};

  ImageData? get(String key) {
    final ref = _cache[key];
    if (ref == null) return null;
    final target = ref.target;
    if (target == null) {
      _cache.remove(key);  // Объект уже собран GC
    }
    return target;
  }

  void put(String key, ImageData image) {
    _cache[key] = WeakReference(image);
  }
}
```

```dart
// Finalizer — callback при сборке GC
final _finalizer = Finalizer<String>((resourceId) {
  print('Cleaning up resource: $resourceId');
  // Освобождение внешнего ресурса
});

class ManagedResource {
  final String id;

  ManagedResource(this.id) {
    _finalizer.attach(this, id, detach: this);
  }

  void dispose() {
    _finalizer.detach(this);
    // Ручное освобождение
  }
}
```

## 7. GC-friendly практики

```dart
// 1. Используйте const для неизменяемых объектов
const defaultConfig = Config(timeout: 30, retries: 3);

// 2. Предпочитайте short-lived объекты
// Young space GC очень быстр для объектов, умирающих молодыми

// 3. Избегайте финализаторов где возможно
// Финализированные объекты живут дольше (минимум +1 GC цикл)

// 4. Используйте typed data для больших массивов
final data = Float64List(100000);  // Один объект, не 100000 boxed Double

// 5. Отменяйте подписки и таймеры в dispose()
// 6. Ограничивайте размер кэшей
// 7. Используйте WeakReference для кэшей необязательных данных
```

## 8. Распространённые ошибки

### ❌ Ручной вызов GC в production

```dart
// В Dart нет публичного API для принудительного GC в production.
// DevTools позволяет это только для отладки.
// GC сам определяет оптимальный момент сборки.
```

### ❌ Надежда на Finalizer для критических ресурсов

```dart
// Finalizer вызывается НЕДЕТЕРМИНИРОВАННО.
// Может быть вызван через секунды, минуты, или никогда (при выходе из VM).

// Для критических ресурсов (файлы, соединения) — всегда dispose() явно.
```
