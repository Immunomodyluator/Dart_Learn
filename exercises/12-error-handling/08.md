# Задание 08 (12.4): Result-тип вместо исключений — 🐛 Найти баг

**Тема:** 12. Обработка ошибок | **Подтема:** 12.4 Result-тип вместо исключений

---

Код должен безопасно обрабатывать ошибки. Найди все проблемы.

```dart
Future<String> loadFromNetwork(String url) async {
  if (url.isEmpty) throw Exception('empty url');
  return 'data from $url';
}

Future<String?> safeLoad(String url) async {
  try {
    return await loadFromNetwork(url);
  } catch (e) {    // bug 1 — перехватываем всё, включая Error (StackOverflow и т.п.)
    // bug 2 — ошибка проглочена, вернём null и потеряем информацию об ошибке
  }
  return null;
}

void processResult(String? data) {
  // bug 3 — нет проверки на null перед использованием
  print(data.toUpperCase());
}

Future<void> main() async {
  final data = await safeLoad('');
  processResult(data);    // NPE здесь

  // «улучшенная» версия с Result — но тоже с багом
  try {
    final result = await loadFromNetwork('good-url');
    print(result);
  } on Exception {
    // bug 4 — блок catch пустой, ошибка проглочена без логирования
  } on Error {
    // bug 5 — Error нельзя ловить в бизнес-логике: это программные ошибки
    print('Error caught, continuing...');
  }
}
```

<summary>Решение</summary>

```dart
Future<String> loadFromNetwork(String url) async {
  if (url.isEmpty) throw Exception('empty url');
  return 'data from $url';
}

// fix: возвращаем Result или бросаем, не глотаем ошибку
Future<String?> safeLoad(String url) async {
  if (url.isEmpty) return null;   // предусловие — явно, не через catch
  try {
    return await loadFromNetwork(url);
  } on Exception catch (e) {     // fix 1: только Exception, не Error
    print('Load failed: $e');    // fix 2: логируем, не глотаем
    return null;
  }
  // Error (StackOverflow, OutOfMemory) пусть распространяется — это баги!
}

void processResult(String? data) {
  if (data == null) {            // fix 3: проверяем null
    print('No data available');
    return;
  }
  print(data.toUpperCase());
}

Future<void> main() async {
  final data = await safeLoad('');
  processResult(data);           // No data available

  final data2 = await safeLoad('good-url');
  processResult(data2);          // DATA FROM GOOD-URL

  try {
    final result = await loadFromNetwork('good-url');
    print(result);
  } on Exception catch (e) {    // fix 4: логируем ошибку
    print('Exception: $e');
  }
  // fix 5: блок on Error убран — Error это программные баги (NullPointerError,
  // StackOverflowError), их не ловят в бизнес-логике
}
```

**Пять багов:** 1) `catch (e)` без `on Exception` ловит и `Error` — программные критические ошибки, которые нужно дать упасть. 2) Пустой `catch` «глотает» ошибку — теряется информация о причине сбоя. 3) Использование `String?` без null-проверки перед методом вызовет `Null check operator used on a null value`. 4) Пустой `on Exception` — снова глотание ошибки; всегда логируй хотя бы `e.toString()`. 5) Перехватывать `Error` в бизнес-логике — антипаттерн: `Error` сигнализирует о баге в коде, а не о штатной ситуации.
