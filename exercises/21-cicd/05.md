# Задание 05 (21.3): Контейнеризация и деплой — 🖊 Написать код

**Тема:** 21. CI/CD и развёртывание | **Подтема:** 21.3 Контейнеризация и деплой

---

Напиши конфигурацию контейнеризации и деплоя для Dart-сервера на основе Shelf.

**Требования:**

1. **`Dockerfile`** — многоэтапная (multi-stage) сборка:
   - Этап `build`: образ `dart:stable`, копирование `pubspec.*`, установка зависимостей, копирование исходников, компиляция в нативный бинарник `dart compile exe`.
   - Этап `runtime`: минимальный образ `debian:bookworm-slim`, копирование только бинарника из `build`-этапа, добавление non-root пользователя, `EXPOSE`, `HEALTHCHECK`, `CMD`.

2. **`docker-compose.yml`** — сервисы `app` (наш сервер) и `redis` (кэш); переменные окружения через `.env`; политика рестарта; healthcheck для `app`.

3. **`Makefile`** (или `tool/docker.dart`) — команды `build`, `push`, `deploy` с тегированием по git-sha.

4. **`.github/workflows/deploy.yml`** — CI/CD пайплайн: сборка образа, пуш в GHCR, деплой через SSH.

```
Структура:
├── Dockerfile
├── docker-compose.yml
├── .env.example
├── Makefile
└── .github/workflows/deploy.yml
```

```dockerfile
# Dockerfile — многоэтапная сборка

# ── Этап 1: сборка ────────────────────────────────────────────────
FROM dart:stable AS build

WORKDIR /app

# Копируем манифесты ПЕРВЫМИ — кэш слоёв не инвалидируется при смене кода
COPY pubspec.yaml pubspec.lock ./
RUN dart pub get

# Теперь копируем исходники
COPY . .

# Компилируем в нативный бинарник (AOT — не требует Dart VM в runtime)
RUN dart compile exe bin/server.dart -o bin/server

# ── Этап 2: минимальный runtime-образ ─────────────────────────────
FROM debian:bookworm-slim AS runtime

# Зависимости для нативного бинарника (libssl, ca-certificates)
RUN apt-get update && apt-get install -y --no-install-recommends \
    libssl3 \
    ca-certificates \
  && rm -rf /var/lib/apt/lists/*

# Non-root пользователь — принцип минимальных привилегий
RUN useradd --create-home --shell /bin/bash appuser
WORKDIR /home/appuser/app

# Копируем только скомпилированный бинарник
COPY --from=build --chown=appuser:appuser /app/bin/server ./server

USER appuser

EXPOSE 8080

# Healthcheck — Docker/Kubernetes будут знать о состоянии контейнера
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

CMD ["./server"]
```

```yaml
# docker-compose.yml
version: "3.9"

services:
  app:
    build: .
    image: dart-server:${VERSION:-latest}
    ports:
      - "${APP_PORT:-8080}:8080"
    environment:
      - PORT=8080
      - REDIS_URL=redis://redis:6379
      - DATABASE_URL=${DATABASE_URL}
      - APP_ENV=${APP_ENV:-production}
    env_file:
      - .env
    depends_on:
      redis:
        condition: service_healthy
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes
    volumes:
      - redis_data:/data
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5

volumes:
  redis_data:
```

```makefile
# Makefile
REGISTRY  = ghcr.io
NAMESPACE = my-org
IMAGE     = dart-server
GIT_SHA   = $(shell git rev-parse --short HEAD)
VERSION   ?= $(GIT_SHA)

.PHONY: build push deploy clean

build:
	docker build \
	  --tag $(REGISTRY)/$(NAMESPACE)/$(IMAGE):$(VERSION) \
	  --tag $(REGISTRY)/$(NAMESPACE)/$(IMAGE):latest \
	  .

push: build
	docker push $(REGISTRY)/$(NAMESPACE)/$(IMAGE):$(VERSION)
	docker push $(REGISTRY)/$(NAMESPACE)/$(IMAGE):latest

deploy: push
	ssh deploy@$(DEPLOY_HOST) " \
	  docker pull $(REGISTRY)/$(NAMESPACE)/$(IMAGE):$(VERSION) && \
	  docker-compose -f /app/docker-compose.yml up -d --no-deps app \
	"

clean:
	docker rmi $(REGISTRY)/$(NAMESPACE)/$(IMAGE):$(VERSION) || true
```

```yaml
# .github/workflows/deploy.yml
name: Build & Deploy

on:
  push:
    branches: [main]
    tags: ["v*"]

env:
  REGISTRY: ghcr.io
  IMAGE: ${{ github.repository }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write # Нужно для пуша в GHCR

    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}

    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }} # Встроенный токен, не нужны доп. секреты

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE }}
          tags: |
            type=sha,prefix=sha-
            type=ref,event=branch
            type=semver,pattern={{version}}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha # GitHub Actions cache для слоёв
          cache-to: type=gha,mode=max

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production # Требует ручного апрува в GitHub Environments

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: deploy
          key: ${{ secrets.DEPLOY_KEY }}
          script: |
            docker pull ${{ env.REGISTRY }}/${{ env.IMAGE }}:sha-${{ github.sha }}
            cd /app
            VERSION=sha-${{ github.sha }} docker-compose up -d --no-deps app
            docker system prune -f
```

**Ключевые практики:** многоэтапная сборка уменьшает финальный образ с ~500 MB до ~30 MB; копирование `pubspec.*` раньше исходников кэширует зависимости между сборками; non-root пользователь снижает риск при компрометации; `depends_on: condition: service_healthy` гарантирует порядок запуска; встроенный `GITHUB_TOKEN` для GHCR не требует настройки дополнительных секретов.
