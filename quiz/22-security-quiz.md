### Вопрос 1 (22.1): SQL-инъекции

Почему следующий код уязвим, и как его исправить?

```dart
// УЯЗВИМО:
final result = await db.query(
  'SELECT * FROM users WHERE name = "${userInput}"',
);
```

- A) Код не уязвим, Dart автоматически экранирует строки
- B) Уязвимость SQL-инъекции: пользователь может вставить `" OR "1"="1` и получить доступ ко всем записям. Исправление — параметризованные запросы: `db.query('SELECT * FROM users WHERE name = ?', [userInput])`
- C) Проблема только в производительности: конкатенация строк медленнее
- D) Уязвимость только при использовании PostgreSQL, SQLite не подвержен

<details>
<summary>Правильный ответ</summary>

**Ответ: B) SQL-инъекция; исправление — параметризованные запросы**

String interpolation подставляет ввод пользователя напрямую в SQL. Атакующий вводит `"; DROP TABLE users; --` и ломает базу данных. Параметризованные запросы (prepared statements) передают данные отдельно от SQL-команды — драйвер БД корректно экранирует параметры, не позволяя им стать частью команды. Это основное правило безопасности при работе с БД.
</details>

### Вопрос 2 (22.2): Управление секретами

Как правильно получать секреты (API-ключи, пароли) в Dart-приложении?

- A) Хранить в `pubspec.yaml` в секции `environment`
- B) Читать из переменных окружения `Platform.environment['SECRET_KEY']` и валидировать при старте приложения — не хранить в коде или в репозитории
- C) Хардкодить в `const String apiKey = 'abc123'` — код всё равно компилируется в бинарник
- D) Шифровать в `lib/config.dart` с помощью `dart:crypto`

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Читать из переменных окружения и валидировать при старте**

Секреты никогда не должны попадать в репозиторий. Правильный паттерн: в продакшне секреты передаются через env-переменные (Docker secrets, Kubernetes ConfigMap/Secret, CI/CD vault). `Platform.environment['DB_PASSWORD']` читает их в рантайме. Валидация при старте (`if (key == null) throw StateError(...)`) предотвращает запуск без обязательной конфигурации.
</details>

### Вопрос 3 (22.3): Безопасность FFI

Как правильно проверить результат нативной функции, которая может вернуть null-указатель?

```dart
final Pointer<Uint8> buffer = nativeAlloc(size);
// Что нужно проверить перед использованием?
```

- A) Ничего — Dart автоматически проверяет нативные указатели
- B) Проверить `buffer.address != 0` (или `buffer == nullptr`) перед разыменованием; никогда не разыменовывать нулевой указатель — это приведёт к segfault и аварийному завершению
- C) Достаточно обернуть в `try/catch` — Dart поймает нативную ошибку
- D) Нативные функции в Dart никогда не возвращают null

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Проверить `buffer.address != 0` перед разыменованием**

Нативный код C/C++ сигнализирует об ошибке выделения памяти через null-указатель (адрес 0). В отличие от Dart-объектов, null-check система Dart не защищает FFI-указатели. Разыменование нулевого указателя вызывает segmentation fault — нарушение защиты памяти, которое немедленно крашит процесс и не поддаётся перехвату через `try/catch`.
</details>
