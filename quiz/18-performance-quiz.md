### Вопрос 1 (18.1): Профилирование с Dart DevTools

Какой инструмент в Dart DevTools используется для поиска узких мест в CPU?

- A) Memory profiler → Allocation Tracing
- B) CPU Profiler → Flame Chart / Bottom Up — показывает, где тратится процессорное время
- C) Network tab → Request Timeline
- D) Logging view → Log levels

<details>
<summary>Правильный ответ</summary>

**Ответ: B) CPU Profiler → Flame Chart / Bottom Up**

CPU Profiler в Dart DevTools делает выборку стека вызовов с заданным интервалом. Flame Chart визуализирует время выполнения иерархически (сверху вниз — цепочка вызовов). Bottom Up показывает «самые горячие» функции по суммарному времени. Запустить: `dart --observe myapp.dart` или встроено в Flutter DevTools.
</details>

### Вопрос 2 (18.2): const и ленивые коллекции

Каков результат и чем он отличается от `list.map().toList()`?

```dart
final iterable = [1, 2, 3, 4, 5].map((x) => x * 2);
print(iterable.first);
print(iterable.first);
```

- A) `2`, `2` — результаты вычислены один раз и закешированы
- B) `2`, `2` — `map()` возвращает ленивый `Iterable`, каждый вызов `first` вычисляет первый элемент заново
- C) Ошибка компиляции: `map()` нельзя использовать без `.toList()`
- D) `[2, 4, 6, 8, 10]` — итерация происходит при первом доступе

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `2`, `2` — `map()` возвращает ленивый `Iterable`, каждый вызов `first` вычисляет первый элемент заново**

`List.map()` возвращает ленивый `MappedIterable` — трансформация выполняется при итерации, не при создании. Каждое обращение к `first` запускает трансформацию заново. Для кеширования результата нужен `.toList()`. Ленивость экономит память при больших коллекциях, но может вести к лишним вычислениям при повторном обходе.
</details>

### Вопрос 3 (18.3): Параллельные Future

В чём разница по производительности между этими двумя вариантами?

```dart
// Вариант A
final a = await fetchA();
final b = await fetchB();

// Вариант B
final results = await Future.wait([fetchA(), fetchB()]);
```

- A) Нет разницы, Dart однопоточный
- B) Вариант A последователен: сначала ждёт `fetchA`, потом `fetchB`. Вариант B запускает оба одновременно и ждёт завершения обоих — быстрее на величину `min(timeA, timeB)`
- C) Вариант B медленнее из-за overhead `Future.wait`
- D) Вариант B доступен только в Dart 3

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Вариант A последователен; вариант B запускает оба одновременно**

В варианте A `fetchB` не запускается, пока не завершится `fetchA` — общее время `timeA + timeB`. В варианте B оба `Future` стартуют немедленно; `Future.wait` ждёт завершения обоих — общее время `max(timeA, timeB)`. При независимых операциях (HTTP, I/O) `Future.wait` значительно ускоряет выполнение.
</details>

### Вопрос 4 (18.4): Сборщик мусора

Как работает garbage collector в Dart VM?

- A) Один генерационный sweep для всех объектов
- B) Двухгенерационная схема: Scavenger (minor GC) быстро очищает новые короткоживущие объекты в пространстве new; Mark-Sweep-Compact (major GC) обрабатывает долгоживущие объекты в old space
- C) Подсчёт ссылок (reference counting) как в Swift/Python
- D) GC не существует в AOT-скомпилированных программах

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Двухгенерационная схема: Scavenger (minor GC) + Mark-Sweep-Compact (major GC)**

Dart VM использует генерационный GC. Большинство объектов «умирают молодыми» — Scavenger эффективно очищает new space с минимальными паузами через copying. Выжившие объекты перемещаются в old space, где работает полный Mark-Sweep с опциональной компактизацией. AOT-программы также используют GC для управления кучей.
</details>
