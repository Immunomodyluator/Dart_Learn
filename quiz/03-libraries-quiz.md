### Вопрос 1 (3.1): import, export, show и hide

Как импортировать из `package:http/http.dart` только класс `Client`, скрыв всё остальное?

- A) `import 'package:http/http.dart' only Client;`
- B) `import 'package:http/http.dart' show Client;`
- C) `import 'package:http/http.dart' hide Client;`
- D) `import 'package:http/http.dart' as http show Client;`

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `import 'package:http/http.dart' show Client;`**

`show` позволяет импортировать только перечисленные имена. `hide` — наоборот, импортирует всё кроме указанного. `only` — несуществующий модификатор. Вариант D синтаксически неверен: `as` и `show` не используются одновременно через пробел.
</details>

### Вопрос 2 (3.2): Deferred imports

Для чего используется ключевое слово `deferred as` при импорте?

- A) Чтобы скрыть весь экспорт библиотеки и открывать его вручную
- B) Чтобы библиотека загружалась лениво — только при первом вызове `loadLibrary()`
- C) Чтобы переименовать библиотеку без создания алиаса
- D) Чтобы запретить повторную инициализацию синглтонов из библиотеки

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Чтобы библиотека загружалась лениво — только при первом вызове `loadLibrary()`**

`deferred as` сообщает компилятору, что загружать код библиотеки нужно только по требованию. Код библиотеки недоступен до вызова `await prefix.loadLibrary()`. Это оптимизация для уменьшения начального времени запуска, особенно актуальная для веб-приложений.
</details>

### Вопрос 3 (3.3): part, part of и директива library

В чём основное назначение директив `part` и `part of`?

- A) `part` подключает внешний пакет; `part of` объявляет, что файл является заглушкой
- B) `part` разбивает одну библиотеку на несколько файлов, которые объединяются во время компиляции; `part of` указывает принадлежность файла к родительской библиотеке
- C) `part of` позволяет файлу наследоваться от другого файла без импорта
- D) `part` эквивалентен `import` с `show *`

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `part` разбивает одну библиотеку на несколько файлов, которые объединяются во время компиляции; `part of` указывает принадлежность файла к родительской библиотеке**

`part`/`part of` — механизм разделения одной логической библиотеки на физически разные файлы. Все part-файлы видят приватные (`_`) идентификаторы друг друга. Именно поэтому сгенерированные файлы (`*.g.dart` от `build_runner`) используют эту директиву — им нужен доступ к приватным членам исходного класса.
</details>
