### Вопрос 1 (9.1): Классы и объекты

Что выведет этот код?

```dart
class Counter {
  int value = 0;
}

void increment(Counter c) {
  c.value++;
}

void main() {
  final c = Counter();
  increment(c);
  print(c.value);
}
```

- A) `0` — объект передаётся по значению, оригинал не изменяется
- B) `1` — объект передаётся по ссылке, мутация видна снаружи
- C) Ошибка компиляции: `final` нельзя передать в функцию
- D) `1`, но только потому что `Counter` — ссылочный тип

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `1` — объект передаётся по ссылке, мутация видна снаружи**

В Dart объекты (классы) передаются по ссылке. Функция `increment` получает ссылку на тот же объект `Counter`, что создан в `main`. Изменение `c.value` внутри функции отражается в исходном объекте. `final` запрещает лишь переприсвоение переменной `c`, но не мутацию объекта.
</details>

### Вопрос 2 (9.2): Конструкторы и именованные конструкторы

В чём преимущество сокращённого синтаксиса `this.field` в конструкторе?

```dart
class Point {
  final double x;
  final double y;
  // Вариант A
  Point(this.x, this.y);
  // Вариант B
  Point(double x, double y) : this.x = x, this.y = y;
}
```

- A) Вариант A компилируется быстрее, чем B
- B) Вариант A — синтаксический сахар для B; они семантически эквивалентны
- C) Вариант A нельзя использовать с `final` полями
- D) Вариант B — единственный способ инициализировать `final` поля

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Вариант A — синтаксический сахар для B; они семантически эквивалентны**

`this.x` в списке параметров конструктора — «initializing formal parameter»: автоматически объявляет параметр с тем же именем и присваивает его полю. Оба варианта эквивалентны, но A короче и предпочтителен по style guide.
</details>

### Вопрос 3 (9.3): Геттеры и сеттеры

Что выведет этот код?

```dart
class Circle {
  double radius;
  Circle(this.radius);
  double get area => 3.14159 * radius * radius;
  set diameter(double d) => radius = d / 2;
}

void main() {
  final c = Circle(5);
  c.diameter = 20;
  print(c.radius);
  print(c.area.toStringAsFixed(2));
}
```

- A) `5.0` и `78.54`
- B) `10.0` и `314.16`
- C) `10.0` и `78.54`
- D) Ошибка компиляции: `final` объект не допускает сеттеры

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `10.0` и `314.16`**

`c.diameter = 20` вызывает сеттер, который устанавливает `radius = 20 / 2 = 10.0`. Затем `area = 3.14159 * 10 * 10 ≈ 314.16`. `final c` означает, что переменная `c` не может быть переприсвоена — но поля объекта изменять можно (если они не `final`).
</details>

### Вопрос 4 (9.4): Наследование и super

Что выведет этот код?

```dart
class Animal {
  String speak() => '...';
}

class Dog extends Animal {
  @override
  String speak() => 'Woof! ${super.speak()}';
}

void main() {
  print(Dog().speak());
}
```

- A) `Woof!`
- B) `Woof! ...`
- C) `...`
- D) Ошибка компиляции: нельзя вызывать `super` в стрелочной функции

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `Woof! ...`**

`super.speak()` вызывает метод родительского класса `Animal`, который возвращает `'...'`. Метод `Dog.speak()` возвращает конкатенацию `'Woof! '` и результата `super.speak()`, что даёт `'Woof! ...'`. `super` доступен в любом виде методов, включая стрелочные.
</details>

### Вопрос 5 (9.5): Абстрактные классы и интерфейсы

В чём разница между `abstract class` и `interface class` в Dart 3?

- A) Нет разницы — оба запрещают прямое создание экземпляров
- B) `abstract class` можно расширять через `extends` и реализовывать через `implements`; `interface class` можно только `implements` — наследование через `extends` запрещено
- C) `interface class` может содержать реализации методов; `abstract class` — только объявления
- D) `abstract class` доступен только внутри пакета; `interface class` — публичный API

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `abstract class` можно расширять через `extends` и реализовывать через `implements`; `interface class` можно только `implements` — наследование через `extends` запрещено**

Модификатор `interface` (Dart 3) запрещает наследование (`extends`) за пределами библиотеки — только реализацию (`implements`). `abstract class` без модификаторов разрешает и `extends`, и `implements`. `interface class` может содержать реализации по умолчанию, которые реализаторы обязаны переопределить.
</details>

### Вопрос 6 (9.6): Mixins и extension methods

Что такое mixin в Dart и как он отличается от обычного наследования?

- A) Mixin — это синоним `abstract class`; разница только синтаксическая
- B) Mixin позволяет добавить поведение к классу без наследования; можно применять к нескольким несвязанным классам через `with`
- C) Mixin — это интерфейс с реализацией всех методов по умолчанию
- D) Mixin работает только для добавления полей; методы через него добавить нельзя

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Mixin позволяет добавить поведение к классу без наследования; можно применять к нескольким несвязанным классам через `with`**

`mixin` в Dart — механизм повторного использования кода. Класс может применить несколько миксинов через `with`, получая их поля и методы без иерархии наследования. Это решает проблему diamond problem и позволяет компоновать поведение (например, `Serializable`, `Loggable`, `Cacheable`) ортогонально иерархии классов.
</details>

### Вопрос 7 (9.7): Extension types (Dart 3.3)

Что выведет код и зачем здесь нужен extension type?

```dart
extension type Meters(double value) {
  Meters operator +(Meters other) => Meters(value + other.value);
}

void main() {
  final a = Meters(1.5);
  final b = Meters(2.5);
  print((a + b).value);
}
```

- A) Ошибка компиляции: extension type не поддерживает операторы
- B) `4.0` — extension type обеспечивает типобезопасность без аллокации
- C) `4.0` — но под капотом создаётся новый объект `Meters` в heap
- D) `1.5` — потому что extension type не копирует значения

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `4.0` — extension type обеспечивает типобезопасность без аллокации**

Extension type — zero-cost wrapper: компилятор полностью стирает тип и работает напрямую с `double`. Никакого дополнительного allocation нет. При этом статическая система типов не позволит смешать `Meters` с `double` напрямую. Это идеально для unit-типов (метры/килограммы), ID (UserId vs OrderId) и т.п.
</details>

### Вопрос 8 (9.8): Статические члены и фабричные конструкторы

Когда предпочтительно использовать `factory` конструктор вместо обычного?

- A) `factory` нужен только для создания синглтонов
- B) `factory` используется когда нужно: вернуть кэшированный объект, вернуть подтип, или выполнить логику до создания
- C) `factory` — обязательный синтаксис для неизменяемых классов
- D) `factory` нельзя использовать в наследниках

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `factory` используется когда нужно: вернуть кэшированный объект, вернуть подтип, или выполнить логику до создания**

`factory` конструктор не обязан создавать новый объект — он может вернуть уже существующий (паттерн singleton/cache), вернуть экземпляр подкласса (паттерн factory method), или выполнить валидацию до создания. В обычном конструкторе `this` инициализируется до выполнения тела, а `factory` — полноценная фабричная функция.
</details>

### Вопрос 9 (9.9): Sealed classes и exhaustiveness (Dart 3)

Зачем использовать `sealed class` вместо `abstract class`?

- A) Sealed class работает быстрее в AOT компиляции
- B) Sealed class ограничивает иерархию одним файлом и позволяет компилятору требовать исчерпывающую обработку всех подтипов в `switch`
- C) Sealed class — синоним `final class`; нельзя ни наследовать, ни реализовывать
- D) Sealed class автоматически добавляет `==` и `hashCode`

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Sealed class ограничивает иерархию одним файлом и позволяет компилятору требовать исчерпывающую обработку всех подтипов в `switch`**

`sealed` означает: все подтипы должны быть определены в том же файле (library). Благодаря этому компилятор знает полный список вариантов и требует обработки каждого в `switch` — если ветка пропущена, это ошибка компиляции, а не runtime баг. Идеально для ADT (Result, Option, Event и т.п.).
</details>

### Вопрос 10 (9.10): Модификаторы классов (Dart 3)

Что запрещает модификатор `base` у класса?

- A) Создание экземпляров класса напрямую через `ClassName()`
- B) Реализацию класса через `implements` за пределами его библиотеки
- C) Расширение класса через `extends`
- D) Использование класса как mixin

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Реализацию класса через `implements` за пределами его библиотеки**

`base class` запрещает `implements` вне пакета/библиотеки, разрешая только `extends`. Это гарантирует, что конструкторы базового класса всегда вызываются (нельзя создать «поддельную» реализацию, обходящую инварианты). `extends` при этом разрешён — наследование с вызовом `super` сохраняется.
</details>

### Вопрос 11 (9.11): Аннотации и метаданные

Для чего предназначена аннотация `@override`?

- A) Без `@override` переопределение метода не работает — это обязательный синтаксис
- B) `@override` — подсказка компилятору: если в суперклассе нет метода с таким именем, выдать предупреждение; помогает ловить опечатки
- C) `@override` делает метод виртуальным, без него диспетчеризация статическая
- D) `@override` необходим только при переопределении абстрактных методов

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `@override` — подсказка компилятору: если в суперклассе нет метода с таким именем, выдать предупреждение; помогает ловить опечатки**

В Dart все методы виртуальны по умолчанию — `@override` не включает и не отключает полиморфизм. Аннотация сигнализирует анализатору, что метод должен существовать в суперклассе; если его там нет (например, опечатка в имени), `dart analyze` сообщит об ошибке. Хорошая практика — всегда ставить при переопределении.
</details>
