### Вопрос 1 (19.1): Структура проекта

В чём разница между «feature-first» и «layer-first» организацией проекта?

- A) Нет разницы — это одно и то же
- B) Layer-first: `lib/data/`, `lib/domain/`, `lib/presentation/` — все фичи разбиты по слоям. Feature-first: `lib/features/auth/`, `lib/features/profile/` — каждая фича содержит свои слои внутри. Feature-first лучше масштабируется при росте числа фич
- C) Feature-first используется только во Flutter, layer-first только в backend
- D) Layer-first — современный подход, feature-first считается антипаттерном

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Layer-first разбивает по техническим слоям; feature-first группирует по бизнес-фичам**

При layer-first все репозитории лежат в `data/`, все модели в `domain/` — при росте проекта приходится прыгать между папками для одной фичи. Feature-first держит всё связанное с фичей рядом (`auth/data/`, `auth/domain/`, `auth/ui/`), что облегчает навигацию и изолированную разработку.
</details>

### Вопрос 2 (19.2): Dependency Injection

В чём главное отличие Dependency Injection (DI) через конструктор от Service Locator (`get_it`)?

- A) Service Locator быстрее в runtime
- B) Constructor DI: зависимости явны в сигнатуре конструктора — их видно при создании класса, легко мокировать в тестах. Service Locator: зависимости скрыты внутри класса (`GetIt.instance.get<Foo>()`), снижает тестируемость
- C) `get_it` не работает в чистом Dart, только во Flutter
- D) Constructor DI невозможен при круговых зависимостях, а `get_it` их решает

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Constructor DI: зависимости явны; Service Locator: зависимости скрыты внутри класса**

Constructor injection — предпочтительный подход для сложной бизнес-логики: зависимости документируют себя через тип, их легко заменить в тестах без магии. Service Locator удобен как клей верхнего уровня (composition root), но при злоупотреблении превращает классы в непрозрачные объекты с неявными зависимостями.
</details>

### Вопрос 3 (19.3): Иммутабельность и copyWith

Зачем использовать паттерн `@immutable` + `copyWith` для доменных объектов?

```dart
@immutable
class UserProfile {
  final String name;
  final int age;
  const UserProfile({required this.name, required this.age});
  
  UserProfile copyWith({String? name, int? age}) =>
    UserProfile(name: name ?? this.name, age: age ?? this.age);
}
```

- A) Это обязательное требование Dart для работы с `const`
- B) Иммутабельные объекты безопасны для кеширования, передачи между изолятами и сравнения по значению; `copyWith` позволяет «изменять» объект, создавая новый экземпляр с обновлёнными полями
- C) `copyWith` нужен только для JSON-десериализации
- D) `@immutable` делает объект `const` автоматически

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Иммутабельные объекты безопасны для кеширования, передачи между изолятами и сравнения по значению; `copyWith` позволяет создавать изменённую копию**

Иммутабельность исключает баги с общим состоянием: нельзя случайно изменить объект через ссылку. Изоляты Dart требуют глубокого копирования при передаче — `const`-объекты передаются безопасно. `copyWith` — идиоматический способ «изменить» поле без мутации: старый объект остаётся нетронутым.
</details>

### Вопрос 4 (19.4): Архитектурные антипаттерны

Какой антипаттерн описывает класс, который знает слишком много и делает слишком много?

- A) Anemic Domain Model — доменные объекты без поведения
- B) God Object (God Class) — один класс, который накапливает ответственности всей системы, нарушая SRP
- C) Primitive Obsession — использование примитивных типов вместо value objects
- D) Shotgun Surgery — изменение одной функции требует правок в десятках мест

<details>
<summary>Правильный ответ</summary>

**Ответ: B) God Object — один класс, который накапливает ответственности всей системы**

God Object (или God Class) нарушает принцип единственной ответственности (SRP): класс знает о пользователях, сессиях, базе данных, HTTP и ещё пяти вещах. Это делает код хрупким, нетестируемым и сложным для понимания. Решение — декомпозиция по ответственностям на меньшие, focused классы.
</details>
