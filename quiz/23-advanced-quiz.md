### Вопрос 1 (23.1): AOT vs JIT компиляция

Какое утверждение точно описывает разницу между AOT и JIT компиляцией в Dart?

- A) JIT — для продакшна, AOT — только для разработки
- B) AOT компилирует весь код заранее в нативный машинный код: нет JIT-warmup, предсказуемая производительность, меньший рантайм. JIT компилирует в рантайме с профилированием и спекулятивной оптимизацией — горячий код становится быстрее, но первый запуск медленнее и нужен больший рантайм
- C) AOT и JIT дают одинаковую производительность в долгосрочной перспективе
- D) JIT невозможен в Dart 3 — только AOT

<details>
<summary>Правильный ответ</summary>

**Ответ: B) AOT — заранее в нативный код; JIT — в рантайме с профилированием**

Flutter debug/hot-reload использует JIT: быстрый перекомпиляция при изменениях, но с overhead рантайма. Flutter release/`dart compile exe` использует AOT: tree shaking (TFA), нет JIT overhead, предсказуемый старт. JIT выигрывает на долгоживущих серверных процессах (warmup + speculative inlining), AOT — на мобильных (размер бинарника, startup time).
</details>

### Вопрос 2 (23.2): Вклад в open-source Dart

Какой первый шаг при участии в open-source репозитории Dart (`dart-lang/*`)?

- A) Сразу делать `git clone` и отправлять PR с исправлениями
- B) Подписать Google CLA (Contributor License Agreement), найти issue с меткой `good first issue`, убедиться что задача не занята, обсудить подход в комментарии
- C) Форкнуть репозиторий и добавить свой пакет как зависимость
- D) Написать письмо команде Dart с описанием вклада

<details>
<summary>Правильный ответ</summary>

**Ответ: B) Подписать CLA, найти `good first issue`, обсудить подход**

Google требует подписания CLA перед принятием любого contribution — без этого PR не сможет быть смёрджен. `good first issue` — метка для задач с ограниченным scope, подходящих новичкам. Обсуждение подхода в issue экономит время: может оказаться, что задача уже решается или требует другого решения. PR должен следовать Conventional Commits стандарту.
</details>

### Вопрос 3 (23.3): Оценка качества пакета

Какой инструмент используется для проверки готовности пакета к публикации и расчёта pub.dev score?

- A) `dart analyze --pub-score`
- B) `dart pub global activate pana && pana .` — Pub ANAlysis tool проверяет документацию, тесты, статический анализ, platforms support и выдаёт score до 160 очков
- C) `dart publish --check-score`
- D) `dart test --coverage=100` автоматически проверяет все требования

<details>
<summary>Правильный ответ</summary>

**Ответ: B) `dart pub global activate pana && pana .`**

`pana` (Pub ANAlysis) — тот же инструмент, который использует pub.dev для расчёта score пакета. Он проверяет: dartdoc покрытие, форматирование, анализ, наличие `CHANGELOG.md`/`README.md`/`LICENSE`, поддержку платформ, null safety. Запуск локально перед публикацией показывает, что именно снижает score. Максимум — 160 points (Likes + Pub points + Popularity).
</details>
